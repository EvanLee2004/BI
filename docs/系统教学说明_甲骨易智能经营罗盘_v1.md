## 2. 一笔钱的旅程（全书主线）

我们跟一条假想链路（**数字仅为示意**，非真实客户）：

> 智云项目明细里有一笔「交付额」→ 最终影响页面「税前利润」。

```mermaid
sequenceDiagram
  participant Z as 智云
  participant F as 抓数 fetch_zhiyun
  participant D as 数据/进料口 xlsx
  participant I as 清洗 ingest
  participant DB as 看板.db std_收入明细
  participant P as profit 计算
  participant S as summary 内存字典
  participant R as render 或 API
  participant B as 浏览器

  Z->>F: 登录+翻页接口
  F->>D: 写出 项目明细.xlsx
  I->>D: readers 读取
  I->>I: normalize 归属月/定位键
  I->>DB: 全量重建 std_ 行
  Note over I,DB: 若有 adj_ 调整则重放
  P->>DB: 只读
  P->>P: 收入=交付额/1.06 …
  P->>S: periods 里塞好税前利润
  R->>S: 预渲染 HTML 或 JSON
  B->>R: GET / 或 /api/v1/cockpit
  R->>B: 像素上的「税前利润」
```

**看图要点**：

1. **浏览器从不直接读库**——只跟 HTTP 要结果。  
2. **算账只在 Python**——前端只切换已算好的块。  
3. **进料口是文件夹**——抓数失败时还可沿用上次 xlsx。

后面每章都会说：这一层在这条链上的哪一环。

## 3. 五层架构逐层拆

```mermaid
flowchart TB
  L1[① 抓数层<br/>login_zhiyun / fetch_zhiyun / fetch SMB]
  IN[进料口 数据/]
  L2[② 清洗层<br/>readers → normalize → adjust]
  L3[③ 存储层<br/>SQLite 看板.db]
  L4[④ 计算层<br/>profit.py]
  L5[⑤ 展示层<br/>render + server + static]
  L1 --> IN --> L2 --> L3 --> L4 --> L5
```

| 层 | 代码在哪 | 设计原因（大白话） |
|----|----------|-------------------|
| 抓数 | `src/ingest/fetch_zhiyun.py` 等 | 把别人系统的数**搬回家**；搬不到就用家里旧的 |
| 进料口 | `数据/*.xlsx` | **唯一接缝**：换来源只改搬家方式，后面算账不动 |
| 清洗 | `normalize.py` `adjust.py` | 日期金额对齐；陆总改过的数**重放**不丢 |
| 存储 | `schema.py` `db.py` | 标准事实 vs 人工指令分开；库是**后端私产** |
| 计算 | `profit.py` | 纯函数：同样输入必出同样税前利润 |
| 展示 | `render.py` `server.py` `static/` | 人看的页面；**前端零金额运算** |

**路由**是什么？——浏览器地址里路径与「服务器哪个函数响应」的对应。  
例：打开 `/bu/游戏` → `server.py` 里 `@app.get("/bu/{name}")` 接到，并检查你能不能看「游戏」这个 BU。

### 4.2 一次 HTTP 请求生命周期

**HTTP** = 浏览器和服务器约定好的「要东西/回东西」的规矩（像填快递单）。

```mermaid
sequenceDiagram
  participant 浏览器
  participant FastAPI as FastAPI 路由
  participant 鉴权
  participant 缓存 as summary/HTML 缓存
  浏览器->>FastAPI: GET /api/v1/cockpit
  FastAPI->>鉴权: cookie 会话
  alt 无权限
    鉴权-->>浏览器: 401/403
  else 通过
    FastAPI->>缓存: 取已算好的 summary
    缓存-->>FastAPI: 字典
    FastAPI->>FastAPI: api_v1 只序列化不算账
    FastAPI-->>浏览器: JSON
  end
```

### 4.3 shell fetch

整体页登录后，先拿到很轻的 `static/shell.html`，再 `fetch('/api/v1/cockpit/fragments')` 用 `page.js` 组装驾驶舱 HTML——**像先拿信封再装信**（B-P5 已删旧的整页 view 路径）。BU 页为隔离仍常直接出 HTML。

### 4.4 `/api/v1/*` 与飞书以后

| 端点 | 干嘛 |
|------|------|
| session / login / logout | 登录态 |
| cockpit | 全公司数字 JSON |
| cockpit/bu/{name} | 单 BU |
| cockpit/fragments | 渲染就绪碎片（page.js 组装） |

飞书机器人（规划）：内网会话下 `GET /api/v1/cockpit`，读 `numbers` 里税前利润推送——**不要刮 HTML**。

## 5. 数据库

### 5.1 三类表

```mermaid
flowchart TB
  STD[std_ 标准事实<br/>每次更新可整表重建]
  ADJ[adj_ 调整指令<br/>只追加·重放]
  MAN[manual_ 手填分摊去税预算<br/>重建不清]
  STD -.逻辑关联定位键.-> ADJ
  MAN --> PROFIT[profit 读取]
  STD --> PROFIT
  ADJ --> STD
```

**哲学**：源头系统的数 vs 陆总改的数 vs 陆总填的数——**三本账不能糊成一本**，否则重抓必丢改动。

### 5.2 有哪些表（从 schema 数：**15**）

- std：收入明细、下单、回款、内部译员、费用明细（5）  
- adj：调整记录（1）  
- manual/meta：手填、手填BU、分摊比例、去税率、历史、预算、预算历史、运行日志、配置变更（9）  

详表：`2_设计/08_数据库设计_schema全表.md`。

```mermaid
erDiagram
  std_收入明细 ||--o{ adj_调整记录 : 定位键
  std_费用明细 ||--o{ adj_调整记录 : 定位键
  manual_手填 ||--o{ manual_历史 : 流水
  manual_分摊比例 {
    string 归属月
    string BU
    float 比例
  }
```

**会话（session）**是什么？——你登录后服务器塞给你的「通行证」（本系统用 **cookie** 记在浏览器里），以后请求自动带上，不用每点一下都输密码。

## 7. 每天怎么活着（运维）

```mermaid
stateDiagram-v2
  [*] --> 空闲
  空闲 --> 日更中: 计划任务 scheduled
  空闲 --> 日更中: 管理端立即更新
  日更中 --> 空闲: 成功写缓存
  日更中 --> 黄或红: 抓数降级/失败
  黄或红 --> 空闲: 仍可能保留旧页
  空闲 --> 更新代码: 一键更新 apply
  更新代码 --> 重启: 退出码 42
  重启 --> 空闲: 看门狗拉起
  重启 --> 回滚: 启动即崩且有标记
  回滚 --> 空闲
```

### 排查决策树（精简）

| 现象 | 先查 |
|------|------|
| 页面空白 | 浏览器 F12 控制台/网络 404；管理端是否 Ctrl+F5；重复 id |
| 数字不对 | 管理端体检；该月手填是否 0；调整是否过期疑似；进料口文件日期 |
| 更新失败 | refresh_status；智云账号；台账路径；看门狗是否在跑 |

## 8. 测试是怎么保住这个系统的

```mermaid
flowchart TB
  T1[单测 426 例级] --> T2[golden 数字/骨架]
  T2 --> T3[回归红线 32 周期]
```

| 层 | 防什么 | 真实故事 |
|----|--------|----------|

11. 否则 pull 可能覆盖未提交改动或失败。
