# 数据来源说明

## 数据源1：智云「项目明细」导出

**位置**：智云 → PMS/项目管理 → 项目明细（左侧栏，不是「项目运营」也不是「任务」）

**操作步骤**（跟实际用过的导出弹窗一致）：
1. 打开「项目明细」视图，视图选「财务统筹」
2. 筛选条件：**「整单交付日期」筛「本年」**（⚠ 不要筛"本月"——「累计」口径需要全年数据，只导出本月的话，管道算出来的"本年累计"收入会严重偏低，因为只有本月这几天的数据）
3. 点右上角导出图标 → 弹出「将视图下数据导出为Excel」
4. 勾选**「导出所有字段」**（不要用默认的"导出当前表格显示列的字段"，要保证 `交付额/本币`、`项目成本/本币`、`整单交付日期`、`业务线` 这些字段都在，用"所有字段"最保险）
5. **导出格式：Excel(.xlsx) 或 CSV(.csv) 都可以**——管道按文件后缀自动识别，两种都支持，统一优先用Excel（同目录下两个文件都在时优先读xlsx）
6. 导出后的文件名带时间戳（如 `项目明细_20260702150334.xlsx`），**重命名/覆盖为 `数据输入/项目明细.xlsx`**（用CSV就是 `数据输入/项目明细.csv`，固定文件名，管道认这个名字）

**用到的字段**：`整单交付日期`（收入/成本确认时点）、`交付额/本币`（收入）、`项目成本/本币`（成本）、`业务线`（BU拆分维度）。

**Excel格式的读取方式**：不认死sheet名，只读当前激活的第一个sheet——智云导出的Excel本来就只有一份数据、一个sheet，不像收单台账那样有多年份多sheet的问题。

**⚠ 已知坑：智云导出的xlsx文件`<dimension>`标签是错的，千万别把读取方式改成`read_only=True`**（2026-07-03用真实新导出`项目明细_20260703100600.xlsx`验证时发现）：这份文件真实有4948行数据，但文件内部XML的`<dimension ref="A1"/>`却只声明"A1一个单元格"——这是智云导出工具自身的bug，不是文件损坏。openpyxl在`read_only=True`（流式读取）模式下会信任这个错误声明、只读出1行就停；`io_load.py`现在用的是**完整加载模式**（`openpyxl.load_workbook(path, data_only=True)`，不传`read_only`），会完整扫描`<sheetData>`忽略`<dimension>`声明，所以不受影响——但这是很容易踩的坑，**以后如果有人想为了大文件性能改成`read_only=True`，一定要先拿真实智云导出文件测过行数对不对，不能只看程序没报错就当读对了**。

## 数据源2：收单台账Excel

**来源**：财务服务器上亮晶维护的收单台账（不是本地文件、不是云盘同步版），会议里确认这份表本身就已经在服务器上，直接从服务器复制/下载最新版本即可。

**结构约定**：
- 一年一个sheet，sheet名=年份的字符串（如 `"2026"`），这个惯例已经稳定用了6年（2021~2026都有），管道会自动按**当前年份**去找对应sheet，不用改配置

### 跨年 SOP（收单台账 · 任务书64·E / 明昊拍板）

1. **每年 1 月**：亮晶在收单台账 xlsx 里**新建当年 sheet**（名=年份四位，如 `2027`），表头与上年一致。
2. 管道用 `load_ledger(cfg, str(当前年))` 读 sheet——**无 sheet 会 KeyError 清晰报错**（不会静默 0），见 `tests/test_year2027.py`。
3. **管理端/config 一般不用改**：`period_pin` 保持 `null` 即跟系统年；只有复现历史月才设 `period_pin`。
4. 共享盘路径仍在「设置→台账路径」；文件名仍为 `收单台账.xlsx`（覆盖更新）。
5. **智云侧**：`zhiyun_since=auto` 跨年首抓前，程序会把四源旧 xlsx + 库快照拷入 `数据/年度归档/<旧年>/`（只一次、永久保留，不受 30 天备份滚动清理影响）。
- 关键列：`含税金额`、`收单月份`/`收单日期`、`业务BU`（亮晶已100%填完，2026-07-02验证）、`对应报表大类`（同上）、`预算归属部门`（v7.2新增读取，**软字段**：老台账没这列不报错；部门预算执行后台仍用）、`预算明细费用类型`（v1.1新增读取；期间费用构成「按类别」主口径 + 利润表/图表下钻，如"房租""交通费""办公用品"等约57种细类）
- **列按表头文字定位，不认列号**：`mapping.resolve_ledger_columns()` 每次运行时都重新读表头，按文字找到每个字段实际在第几列——亮晶插列、调整列顺序都不影响；哪怕表头文字本身改了名字（已经发生过一次："业务BU"→"利润归属中心"，同一列换了个名字），只要在 `mapping.py` 的 `LEDGER_COLUMN_ALIASES` 里加一个候选名字就行，不用改列号。**真的一个候选名字都对不上时，程序会直接报错停下**、把它当时读到的完整表头打印出来，不会悄悄拿错列算出一堆好看但错误的数字。
- 下载后**重命名/覆盖为 `数据输入/收单台账.xlsx`**

**分类规则（2026-07-03起，无映射表兜底）**：`业务BU`、`对应报表大类` 这两列由财务按统一规则**逐行直接填写**，管道只读这两列的直接值——没填就标"未分类"并在页面显眼提示，不会查任何映射表去猜。填充进度：截至2026-07-02，两列亮晶已**100%填完**（早前是43%/45%）。此前版本有一张 `reference/费用及报表映射.xlsx` 映射表用于兜底未填的行，2026-07-03起已从管道里移除（不再读取，倒逼分类在收单台账源头一次填对）；该文件仍保留在 `reference/` 目录作历史存档，不影响任何计算。
